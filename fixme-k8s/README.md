# FixMe K8s Setup

## Prerequisites

- Docker Desktop (Windows) with WSL2 integration enabled
- WSL2 distro (Ubuntu 24) with WSLg enabled (default on Windows 11)
- Minikube installed inside WSL2
- Lens installed inside WSL2 (renders on Windows via WSLg)

## Install Minikube (inside WSL2)

```bash
curl -LO https://github.com/kubernetes/minikube/releases/latest/download/minikube-linux-amd64
sudo install minikube-linux-amd64 /usr/local/bin/minikube && rm minikube-linux-amd64
```

## Install Lens (inside WSL2)

```bash
curl -fsSL https://downloads.k8slens.dev/keys/gpg | sudo gpg --dearmor -o /usr/share/keyrings/lens-archive-keyring.gpg
echo "deb [arch=amd64 signed-by=/usr/share/keyrings/lens-archive-keyring.gpg] https://downloads.k8slens.dev/apt/debian stable main" \
  | sudo tee /etc/apt/sources.list.d/lens.list
sudo apt-get update && sudo apt-get install -y lens
```

## Start Minikube

First time only (flags are remembered for subsequent starts):

```bash
minikube start --driver=docker --listen-address=0.0.0.0 --insecure-registry="10.0.0.0/24"
```

After that, just:

```bash
minikube start
```

## Launch Lens

```bash
lens-desktop &
```

Lens reads `~/.kube/config` directly from WSL2. No kubeconfig syncing or TLS workarounds needed.

## Enable Registry Addon

```bash
minikube addons enable registry
```

The registry addon with the docker driver uses a non-standard port (e.g. `59047`) instead of `5000`.
Check the output of `minikube addons enable registry` for the actual port.

## Push Images to Minikube Registry

```bash
docker tag my-image localhost:59047/my-image
docker push localhost:59047/my-image
```

In Kubernetes manifests, reference images as:

```yaml
image: localhost:59047/my-image
```

## After Reboot

```bash
minikube start
lens-desktop &
```

## Deploy Redis to Minikube

Make sure minikube is running, then create the namespace and apply the manifests.
Run these from the `fixme-service` directory:

```bash
cd ~/projects/fixme-service

# Create the production namespace (safe to run if it already exists)
kubectl create namespace production 2>/dev/null; true

# Apply all Redis manifests at once (order doesn't matter — Kubernetes resolves dependencies)
kubectl apply -f fixme-k8s/minikube/fixme-redis/
```

This creates: StorageClass → PersistentVolume → ConfigMap → StatefulSet → Service.

### Verify it's running

```bash
# Pod status (should show Running 1/1)
kubectl get pods -n production

# Detailed pod info and events (scheduling, image pull, startup)
kubectl describe pod fixme-redis-0 -n production

# Redis container logs
kubectl logs fixme-redis-0 -n production

# Init container logs (master/replica discovery)
kubectl logs fixme-redis-0 -n production -c config

# Service and endpoints
kubectl get svc -n production

# Persistent volume and claim status (should show Bound)
kubectl get pv,pvc -n production

# Test Redis connection from inside the cluster
kubectl exec -it fixme-redis-0 -n production -- redis-cli -a thisismyownpassword123@ ping
```

### Troubleshooting

```bash
# Pod stuck in Pending — check events for scheduling or volume issues
kubectl describe pod fixme-redis-0 -n production | tail -20

# Pod in CrashLoopBackOff — check logs for config errors
kubectl logs fixme-redis-0 -n production --previous

# PV/PVC stuck in Terminating — clear finalizers to unstick
kubectl patch pvc data-fixme-redis-0 -n production -p '{"metadata":{"finalizers":null}}'
kubectl patch pv redis-pv -p '{"metadata":{"finalizers":null}}'

# Watch pod status in real time (Ctrl+C to stop)
kubectl get pods -n production -w
```

### Tear down Redis

```bash
# Delete all Redis resources
kubectl delete -f fixme-k8s/minikube/fixme-redis/

# If you also want to remove the PVC (deletes stored data)
kubectl delete pvc data-fixme-redis-0 -n production
```

## Docker Compose vs Kubernetes — How It Maps

In `docker-compose.yaml`, everything is in one file. In Kubernetes, the same concepts are split across multiple manifests. Here's how they map using Redis as an example:

| Concept | Docker Compose | Kubernetes (minikube) |
|---|---|---|
| **What runs the container** | `image: redis:alpine` in docker-compose.yaml | `image: redis:alpine` in statefulSet.yaml |
| **Container name** | `container_name: redis_container` | Pod name: `fixme-redis-0` (auto-generated by StatefulSet) |
| **Networking** | Other containers reach it via `redis_container:6379` on the `elastic` network | Other pods reach it via `fixme-redis.production.svc.cluster.local:6379` using the Service |
| **Port exposure** | `ports: "6379:6379"` maps to host | Service with `ClusterIP` — internal only, no host mapping |
| **Persistent data** | `volumes: ./docker-volumes/cache:/data` — a folder on your machine | PV + PVC — a folder on the minikube node (`/storage/data1`) |
| **Configuration** | `command: redis-server --loglevel warning` | ConfigMap with full `redis.conf`, mounted as a file into the pod |
| **Restart policy** | `restart: always` | Kubernetes does this by default — it always restarts crashed pods |

### How it works in minikube

1. **You apply the manifests** — `kubectl apply -f fixme-redis/`
2. **Kubernetes reads the StatefulSet** and sees it needs 1 pod running `redis:alpine`
3. **The pod needs storage** — the volumeClaimTemplate creates a PVC (`data-fixme-redis-0`), which binds to the PV defined in `pv.yaml`, pointing to `/storage/data1` on the minikube node
4. **The pod needs config** — the ConfigMap (`redis-config`) is mounted as `/tmp/redis/redis.conf`, then the init container copies it to a writable volume at `/etc/redis/redis.conf`
5. **The pod starts** — Redis runs with that config, listening on port 6379
6. **The Service makes it reachable** — other pods connect to `fixme-redis:6379` and the Service routes traffic to the Redis pod

### Why more files?

Kubernetes separates concerns into individual resources:

- **StorageClass** → "what kind of storage" (local disk, cloud SSD, etc.)
- **PersistentVolume (PV)** → "here's the actual disk space"
- **ConfigMap** → "here's the configuration"
- **StatefulSet** → "run this container with this config and this storage"
- **Service** → "make it reachable by name"

The tradeoff is more files, but each piece is independently manageable — you can update the config without restarting the pod, resize storage without redeploying, etc.

### About the container image

The image is the same — `redis:alpine`. Minikube pulls it from Docker Hub just like Docker Compose does. The difference is that minikube has its own Docker daemon inside its container, so `minikube image ls` shows different images than `docker image ls` on your host.

## Notes

- `minikube delete` is required to change startup flags like `--insecure-registry`
- Lens runs inside WSL2 and renders on Windows via WSLg — no IP or kubeconfig issues
- Docker Desktop forwards WSL2 localhost ports to Windows, so `localhost` works from both sides
- See [MINIKUBE-LENS-SETUP.md](MINIKUBE-LENS-SETUP.md) for detailed setup steps and kubectl cheat sheet
